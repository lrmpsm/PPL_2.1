## Лабораторная работа №1 (`35c3f452ce62058b7d46b59a3262804876c9bcba`)
* Тема: Источники задач и контракты

### Реализация

#### Задача

Задача была представлена моделью `Task` (`src/models/task.py`).

Имеет поля:
- `id: str` - идентификатор задачи
- `payload: Any` - данные задачи

#### Источники задач

Реализованы следующие источники задач:
- `generator` - генерация `n` задач со случайным `payload` (`src/sources/generator.py`)
- `stdin` - чтение задач из стандартного ввода (`src/sources/stdin.py`)
- `file-jsonl` - чтение задач из `jsonl`-файла (`src/sources/json.py`)

#### Контракт источников

Контракт `TaskSource` для источников описан с помощью `typing.Protocol` (`src/contracts/task_source.py`).

В контракт входят:
- `name: str` - имя источника
- `fetch()` - метод получения задач

Для возможности runtime-проверки используется `@runtime_checkable`.

Во время получения задач каждый источник проверяется на соответствие контракту (`src/inbox/core.py`).

#### Реестр источников

Для регистрации источников используется `REGISTRY` (`src/sources/repository.py`).

Добавление новых источников выполняется через декоратор `register_source`.

Такой подход позволяет добавлять новые источники без изменения кода `InboxApp`.

#### CLI

В проекте реализован CLI (`src/cli.py`).

Поддерживаются команды:
- `plugins` - вывод списка доступных источников
- `read` - чтение задач из выбранных источников

Для команды `read` можно выбрать источники:
- `--stdin`
- `--jsonl`
- `--generator`

Также поддерживается фильтрация задач по содержимому `payload` через параметр `--contains`.

### Тесты

Написаны unit-тесты для:
- интерфейса командной строки (`tests/test_cli.py`)
- приема задач из нескольких источников (`tests/test_inbox.py`)
- источников задач:
    - генератора (`tests/test_generator.py`)
    - стандартного ввода (`tests/test_stdin.py`)
    - `jsonl`-файлов (`tests/test_json.py`)

Тесты проверяют:
- создание задач из разных источников;
- соответствие источников контракту `TaskSource`;
- runtime-проверку контракта;
- регистрацию источников;
- работу CLI-команд.

### Логирование

В проекте используется стандартный модуль `logging`.

Конфигурация находится в файле `src/logging_config.py`.

Логи записываются в файл `logs/shell.log`.

## Лабораторная работа №2
* Тема: Модель задачи: дескрипторы и `@property`

### Реализация

#### Доменная модель `Task`

Модель задачи была расширена в соответствии с требованиями ЛР №2 (`src/models/task.py`).

Имеет атрибуты публичного API:
- `id: str` - идентификатор задачи
- `description: str` - описание задачи
- `priority: int` - приоритет выполнения задачи
- `status: TaskStatus` - статус задачи
- `created_at: datetime` - время создания
- `updated_at: datetime` - время последнего изменения
- `is_ready: bool` - вычисляемое свойство готовности задачи к выполнению
- `created_at_text` - время создания в более человекочитаемом формате
- `updated_at_text` - время последнего изменения в более человекочитаемом формате

Внутреннее состояние хранится в `protected`-атрибутах:
- `_id`
- `_description`
- `_priority`
- `_status`
- `_created_at`
- `_updated_at`

#### Метод `_touch`

Метод необходим для обновления атрибута `_updated_at` и сброса кэшированного значения `updated_at_text` после изменения атрибутов публичного интерфейса.

#### Модель `TaskStatus`

Класс перечисления для статуса задачи (`src/models/task_status.py`).

Содержит следующие элементы перечисления:
- `CREATED` - задача создана
- `PENDING` - задача в очереди на выполнение
- `IN_PROGRESS` - задача в процессе выполнения
- `FAILED` - задачу не удалось выполнить в связи с ошибкой
- `DONE` - задача выполнена

#### Свойства `Task` (`properties`)

С помощью декоратора `@property` были реализованы следующие свойства модели `Task`:
- `created_at`
    - read-only свойство
- `updated_at`
    - read-only свойство
- `is_ready`
    - read-only свойство
    - вычисляется на основе текущего `status` объекта
- `status`
    - есть setter
    - если значение передано строкой, setter пытается преобразовать его в `TaskStatus`
    - если значение не является строкой или объектом `TaskStatus`, генерируется `InvalidTaskStatusError`

#### Дескрипторы

Все дескрипторы доменной модели задачи находятся по пути `src/models/descriptors.py`.

##### Data Descriptors

Все data descriptors ЛР наследуются от общего класса-дескриптора `ValidatedField`.

В `ValidatedField` определены общие реализации методов:
- `__set_name__()`:
    - формирует имя внутреннего атрибута с нижним подчёркиванием в начале
- `__get__()`:
    - если дескриптор вызывается из класса, возвращает объект этого дескриптора
    - иначе возвращает атрибут соответствующего объекта
- `__set__()`:
    - валидирует присваиваемое значение
    - в случае успешной валидации присваивает его соответствующему объекту
    - если в модели определён метод `_touch`, то он вызывается

На основе `ValidatedField` реализованы следующие дескрипторы:
- `NonEmptyLimitedString`
    - используется атрибутами `id` и `description`
    - проверяет, что значение является строкой, непустое после `strip()` и не превышает максимальную длину
- `Priority`
    - используется атрибутом `priority`
    - проверяет, что значение является целым числом и входит в допустимый диапазон

##### Non-Data Descriptors

`TimeAsText` - non-data descriptor для генерации человекочитаемых атрибутов типа `str` на основе внутренних атрибутов типа `datetime`.

В модели `Task` используется два объекта данного дескриптора:
- `created_at_text`
- `updated_at_text`

#### Исключения

Все пользовательские исключения для доменной модели задачи находятся по пути `src/models/exceptions.py`.

Среди них:
- `TaskError` - базовый класс исключений для модели `Task`
- `InvalidTaskIdError` - для некорректного идентификатора задачи
- `InvalidTaskDescriptionError` - для некорректного описания задачи
- `InvalidTaskPriorityError` - для некорректного приоритета выполнения задачи
- `InvalidTaskStatusError` - для некорректного статуса задачи

Написаны unit-тесты (`tests/test_task.py`), которые проверяют:

## Лабораторная работа №3
* Тема: Очередь задач: итераторы и генераторы

### Реализация

#### Очередь задач `TaskQueue`

Для потоковой обработки задач реализована очередь `TaskQueue` (`src/models/task_queue.py`).

Очередь принимает набор источников задач, соответствующих контракту `TaskSource`, и объединяет их в единый поток задач.

Публичный API очереди:
- `__iter__()` - возвращает итератор по задачам из всех источников
- `filter_by_statuses()` - фильтрует задачи по одному или нескольким статусам
- `filter_by_priority()` - фильтрует задачи по точному значению приоритета
- `filter_by_priority_range()` - фильтрует задачи по диапазону приоритетов

#### Протокол итерации

Класс `TaskQueue` реализует протокол итерации через метод `__iter__()`.

Метод последовательно обходит все переданные источники задач и получает задачи через `source.fetch()`.

#### Повторный обход очереди

`TaskQueue` поддерживает повторную итерацию.

При каждом новом проходе по очереди заново вызывается метод `fetch()` у каждого источника.

#### Ленивые фильтры

Фильтрация задач реализована через генераторы.

Метод `filter_by_statuses()` принимает статусы в виде:
- объектов `TaskStatus`
- строковых значений статусов

Перед фильтрацией строковые значения приводятся к `TaskStatus`.

Если передан неизвестный статус, генерируется пользовательское исключение `InvalidTaskStatusError`.

Методы `filter_by_priority()` и `filter_by_priority_range()` также возвращают генераторы и не создают промежуточные списки задач.

### Тесты

Написаны unit-тесты (`tests/test_task_queue.py`), которые проверяют:
- итерацию по задачам из нескольких источников с сохранением порядка получения элементов
- повторный обход очереди без изменения её состояния и без потери задач
- корректное завершение итерации после получения последнего элемента
- фильтрацию задач по статусу, точному значению приоритета и диапазону приоритетов
- работу с пустой очередью и обработку некорректных значений статуса
